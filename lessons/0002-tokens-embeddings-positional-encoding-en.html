    Tokens, Embeddings, and Positional Encoding

    How is human language converted into mathematical vectors and how are word positions preserved?

    Transformer Block Map — where you are

    Focus of this lesson Rest (other lessons)

    You are here: this lesson covers the entry stage — Input Tokens, Token Embedding and Positional Encoding (highlighted below). The inner Transformer Block comes in later lessons.

    Introduction

    In the previous lesson, we learned that an LLM or Transformer essentially works to predict the next word. But the question is: computers do not understand the text or writing we provide directly. They only understand numbers!

    To send a sentence into a Transformer, it must undergo 3 fundamental processes:

    1. Tokenization: Breaking the sentence down into smaller pieces or tokens and assigning a unique number (ID) to each.
    2. Embedding: Converting the token IDs into a meaningful high-dimensional vector or a list of numbers.
    3. Positional Encoding: Adding positional information to the vector indicating where each word is located in the sentence.

    Let's dive deep into these three concepts using simple explanations and interactive simulations.

    1. Tokenization

    Tokenization is the process of breaking input text down into small, meaningful pieces called Tokens. These tokens can be whole words, sub-words, or even individual characters.

    Key tokenization concepts are outlined below:

    • Vocabulary: This is the model's own dictionary. The total number of unique tokens the model knows is the vocabulary size (e.g., GPT-4's vocab size is around 100,000).
    • Token IDs: After tokenizing, each token is assigned a number or ID corresponding to its index in the dictionary. For example, "cat" might be 3797.

    Why is Sub-word Tokenization used in modern LLMs?

    If we work only with whole words (word-level), the dictionary becomes extremely large, and if there is any unfamiliar word or typo, the model cannot recognize it (known as the Out-of-Vocabulary or OOV problem). On the other hand, if we only use individual characters (character-level, e.g., a, b, c), the token count per sentence becomes very large, making it difficult to process.

    Sub-word algorithms like Byte Pair Encoding (BPE) solve this issue. BPE keeps frequently used words whole while splitting unfamiliar words into familiar sub-words.

    Example of Sub-word Tokenization:

    If the model does not find the word "unhappiness" in its vocabulary, it splits it into three familiar sub-word tokens:
    "un" (ID: 421) + "happi" (ID: 8932) + "ness" (ID: 1205)

    2. Embeddings

    After tokenization, we have token IDs or numbers. However, IDs alone are not enough for mathematical processing. For example, if "cat" has ID 3797 and "dog" has ID 4251, mathematically 4251 > 3797 has no meaning. It does not explain what relationship exists between cats and dogs.

    This is where Embeddings come in. Each token ID is converted into a High-dimensional Dense Vector (a list of numbers). This is called the Embedding Space. In practice, the vector size (dmodel) ranges from 768 up to 4096 or more.

    Semantic Space:

    Each dimension of the embedding vector represents some subtle semantic meaning or grammatical feature (e.g., gender, royalty, animacy, etc.). Words with similar meanings will have vectors positioned close to each other in this multi-dimensional space.

    As a result, we can perform semantic vector math on words! For example:

    Vector("King") - Vector("Man") + Vector("Woman") ≈ Vector("Queen")

    3. Positional Encoding

    We now have word meanings represented as embeddings. However, a unique feature of the Transformer architecture is that it processes all tokens simultaneously in parallel. It does not process words one by one like older RNN or LSTM models.

    While parallel processing makes the model extremely fast, it introduces a new problem: the model loses track of word order or sequence (Permutation Invariance).

    Without positional encoding, the model cannot distinguish between these two sentences:

    1. "The cat ate the mouse"
    2. "The mouse ate the cat"

    Since both sentences contain the exact same tokens, they would look identical to the model without positional information.

    Solution: Adding Positional Encoding Vectors

    For each position in the sentence (Index 0, 1, 2...), a unique mathematical vector of the same size as the embedding vector is constructed. This position vector is then added to the token embedding vector:

    Combined Representation = Token Embedding + Positional Encoding

    Trigonometric sine and cosine waves are used to generate positional vectors, enabling the model to learn relative and absolute word positions.
